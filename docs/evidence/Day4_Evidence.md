# Day4 Evidence - 发布表单与数据新增

## 1. 今日完成内容

Day4 主要完成了发布表单的设计、表单校验逻辑的实现和数据新增功能的开发：

**发布表单设计：**
- 创建了 `src/components/FormField.vue` 通用表单项组件，统一处理标签、必填标识和错误提示
- 设计了 `src/views/PublishView.vue` 发布页面，支持四种发布类型（二手交易、失物招领、拼单搭子、跑腿委托）的动态切换

- 实现了基础表单校验逻辑，包括必填校验和数值范围校验

**数据新增开发：**
- 为 trade.ts、lostFound.ts、groupBuy.ts、errand.ts 四个业务 API 模块补充了 POST 方法
- 将所有接口的 id 字段改为可选（id?: number），由 JSON Server 自动生成
- 实现了数据提交逻辑，提交成功后跳转到对应列表页并显示成功提示
- 实现了表单重置功能，清空所有字段和错误信息

## 2. 发布类型与字段设计

| 发布类型 | 对应数据集合 | 关键字段 | 设计理由 |
|---|---|---|---|
| 二手交易 | trades | title、category、price、condition、location、description | 需要展示商品基本信息和交易条件，price 和 condition 是买家最关心的属性，category 用于分类筛选 |
| 失物招领 | lostFounds | title、type、itemName、location、eventTime、description | 需要区分寻物和招领两种类型，itemName 明确物品名称便于快速识别，eventTime 对失主和捡到者都很重要 |
| 拼单搭子 | groupBuys | title、type、targetCount、deadline、location、description | 需要说明拼单类型、人数目标和截止时间，targetCount 和 deadline 是拼单业务的核心属性 |
| 跑腿委托 | errands | title、taskType、reward、from、to、deadline、description | 需要说明任务内容、起止地点和酬劳，reward 是吸引接单者的关键因素，from 和 to 明确任务范围 |

## 3. 表单校验规则

发布表单设置了以下校验规则：

**通用字段校验：**
- 标题不能为空；
- 地点不能为空；
- 描述不能为空；

**二手交易专属校验：**
- 商品分类不能为空；
- 价格必须大于 0；
- 成色不能为空；

**失物招领专属校验：**
- 物品名称不能为空；
- 发生时间不能为空；

**拼单搭子专属校验：**
- 拼单类型不能为空；
- 目标人数不能少于 2 人；
- 截止时间不能为空；

**跑腿委托专属校验：**
- 任务类型不能为空；
- 酬劳不能为负数；
- 取件地点不能为空；
- 送达地点不能为空；
- 截止时间不能为空；

## 7. 遇到的问题与解决方法

**问题 1：字段名称不匹配**
- 现象：提交数据后字段名称与 db.json 不一致，导致列表无法正确显示新增数据；
- 原因：AI 生成的字段名称与 API 接口定义不匹配（如使用 type 而非 lostFoundType）；
- 解决：对照 API 接口定义和 db.json 数据结构，将 type 字段拆分为 lostFoundType 和 groupType，确保所有字段名称一致。

**问题 2：数字类型校验错误**
- 现象：价格输入 0 时校验通过，但应该提示错误；
- 原因：使用 `!form.price` 判断，0 会被 JavaScript 视为 falsy 值；
- 解决：改为 `form.price <= 0` 进行判断，确保正确区分 0 和空值。

**问题 3：提交后跳转不正确**
- 现象：所有类型提交后都跳转到首页，无法看到发布结果；
- 原因：AI 生成的代码中没有根据发布类型进行条件跳转；
- 解决：在提交逻辑中添加条件判断，根据 publishType 的值跳转到对应列表页（/trade、/lost-found、/group-buy、/errand）。

**问题 4：重置函数没有正确处理数字类型**
- 现象：点击重置按钮后，价格、酬劳等数字字段变为空字符串，导致后续提交失败；
- 原因：使用 `(form as Record<string, unknown>)[key] = ''` 统一重置所有字段；
- 解决：改为逐个字段手动重置，确保数字类型字段设置为正确的默认值（price=0, reward=0, targetCount=2）。

**问题 5：JSON Server 端口占用**
- 现象：启动 npm run mock 时提示端口 3001 已被占用；
- 原因：之前启动的 Mock 服务仍在运行；
- 解决：使用已运行的服务，无需重新启动。

## 8. 今日反思

发布表单、表单校验和数据新增是完整 Web 应用不可或缺的核心功能。发布表单是用户与系统交互的入口，它决定了用户如何输入数据，一个好的表单设计应该简洁明了，字段设置合理，符合用户的使用习惯。表单校验则是保证数据质量的第一道防线，通过校验可以防止用户输入无效数据，避免脏数据进入数据库，同时也能给用户及时的反馈，提升用户体验。数据新增是实现数据持久化的关键步骤，它将用户填写的表单数据发送到后端服务器并存储到数据库中，使数据能够在应用中被其他用户访问和使用。

在设计发布表单时，字段的选择和设计需要紧密结合业务场景。例如二手交易需要价格和成色字段，失物招领需要类型和物品名称字段，拼单搭子需要目标人数和截止时间字段，跑腿委托需要取件地点和送达地点字段。这些字段都是对应业务的核心属性，缺失任何一个都会影响业务流程的完整性。同时，表单的布局和交互也很重要，使用条件渲染根据发布类型显示不同字段，可以避免表单过于冗长，提升用户填写效率。

表单校验不仅要检查字段是否为空，还要根据业务规则进行数值范围和格式校验。例如价格必须大于 0，目标人数不能少于 2 人，酬劳不能为负数等。这些校验规则直接关系到数据的有效性和完整性，如果没有校验，用户可能会提交无效数据，导致后续业务逻辑出错。

数据新增功能需要确保字段名称和类型与后端接口完全匹配，否则会导致提交失败或数据存储错误。同时，提交成功后的反馈和跳转也很重要，应该给用户明确的成功提示，并跳转到对应列表页，让用户能够看到发布结果，增强用户的信心和满足感。

通过今天的实践，我深刻认识到发布表单、表单校验和数据新增三者之间的紧密关系，它们共同构成了 Web 应用的数据输入和存储链路。只有三者都设计合理、实现正确，才能保证应用的正常运行和良好的用户体验。
